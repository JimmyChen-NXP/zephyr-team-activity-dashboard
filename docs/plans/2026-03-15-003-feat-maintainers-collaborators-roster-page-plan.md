# feat: Maintainers & Collaborators roster page

## Overview

Add a new static page at `/maintainers` that displays which team roster members appear in the Zephyr MAINTAINERS.yml file, separated clearly into **Maintainers** and **Collaborators** sections. A maintainer and a collaborator are never shown in the same bucket, but a person can appear in both sections if they hold both roles across different subsystems.

## Problem Statement / Motivation

The dashboard tracks activity metrics but has no view that answers: *"Who on our roster owns which parts of the codebase?"* The Zephyr MAINTAINERS.yml is the authoritative source for subsystem ownership. Surfacing it lets the team quickly identify coverage gaps, onboarding targets, and ownership distribution without leaving the dashboard.

## Proposed Solution

1. **CI data-collection script** ŌĆö a Node.js script (`scripts/collect-maintainers.mjs`) runs during the `deploy` job (or as a new `collect-maintainers` step). It fetches `MAINTAINERS.yml` from `raw.githubusercontent.com`, parses it, cross-references logins against `upstream_member.csv`, and writes `public/maintainers-map.json` to the data branch.
2. **New page** ŌĆö `src/app/maintainers/page.tsx` follows the same GitHub-Pages / Live dual-mode pattern as existing pages. In snapshot mode it fetches `/maintainers-map.json` client-side.
3. **Two-section layout** ŌĆö a `MaintainersPage` component renders a Maintainers table and a separate Collaborators table. They are never merged. A person appears in both tables if they hold both roles (across different subsystems).
4. **Navigation** ŌĆö a site-level nav link "Maintainers" is added to the title bar so users can navigate from any activity page.

## MAINTAINERS.yml Structure

Each entry in the file is a named subsystem block:

```yaml
Bluetooth:
  status: maintained
  maintainers:
    - johanhedberg
    - jhedberg
  collaborators:
    - carles-cufi
    - lapsa
  files:
    - subsys/bluetooth/
  labels:
    - "area: Bluetooth"
```

The script inverts this map: for each login, collect which subsystems list them as `maintainer` vs `collaborator`.

## Data Shape ŌĆö `public/maintainers-map.json`

```json
{
  "generatedAt": "2026-03-15T12:00:00Z",
  "maintainers": [
    {
      "login": "johanhedberg",
      "name": "Johan Hedberg",
      "subsystems": ["Bluetooth", "Bluetooth: HCI Drivers"]
    }
  ],
  "collaborators": [
    {
      "login": "carles-cufi",
      "name": "Carles Cufi",
      "subsystems": ["Bluetooth", "Bluetooth: Classic"]
    }
  ]
}
```

- `name` is resolved from `upstream_member.csv` during generation; falls back to `login` if not in roster.
- Only logins present in the roster CSV are included (the page is about *our team's* roles, not all external contributors).

## Technical Considerations

- **YAML parsing**: use `js-yaml` (add as a dev/build dependency). It is already widely used in Next.js ecosystems and is small.
- **Roster cross-reference**: the script reads `upstream_member.csv` at the project root (same file used by `src/lib/roster.ts`) and builds a `Map<login, name>` before parsing the YAML.
- **No rate-limit risk**: `raw.githubusercontent.com` is public and unauthenticated. One fetch per CI run is well within limits.
- **CI placement**: the script runs in the `deploy` job (before `next build`) so the generated JSON is placed at `public/maintainers-map.json` alongside the sparse-checked-out snapshot files. No new CI job is needed.
- **Static export compatibility**: the page is fully client-rendered (`"use client"`), same as `SnapshotDashboardPage`. No server-side data access at build time.
- **Stale data**: the JSON is regenerated on each deploy. The page shows a `generatedAt` timestamp (same pattern as snapshot pages).

## System-Wide Impact

- **Navigation**: `ActivityPageNav` is activity-view-specific. The "Maintainers" link should be a peer-level nav item in the title bar, not inside `ActivityPageNav`. Either extend the title bar or add a small `SiteNav` component that wraps both the activity links and the maintainers link.
- **No snapshot data dependency**: the maintainers page does NOT require a snapshot to be loaded. It fetches its own JSON. This keeps load time independent.
- **Data branch**: `collect-and-deploy.yml` must copy `public/maintainers-map.json` from the deploy workspace to the data branch, similar to how snapshots are handled. Alternatively, generate the file inside the deploy job after the sparse checkout (so it sits next to the snapshot files naturally).

## Acceptance Criteria

- [ ] `public/maintainers-map.json` is generated by a CI script and stored on the data branch
- [ ] Script only includes roster members (logins present in `upstream_member.csv`)
- [ ] Page at `/maintainers` is reachable from the title-bar nav on all pages
- [ ] Page renders a "Maintainers" section and a separate "Collaborators" section ŌĆö never combined
- [ ] Each row shows: display name (from roster), GitHub login (linked to `github.com/<login>`), and a list of their subsystem names
- [ ] A person who is both maintainer (in some subsystems) and collaborator (in others) appears in **both** sections
- [ ] Subsystem list is sorted alphabetically per person
- [ ] Page works on GitHub Pages (static snapshot mode) and shows a `generatedAt` timestamp
- [ ] Empty state shown if JSON fails to load
- [ ] No new runtime errors on existing activity pages

## Dependencies & Risks

| Item | Notes |
|---|---|
| `js-yaml` npm package | Add to `dependencies`; very stable |
| `raw.githubusercontent.com` accessibility from CI | Standard public URL; should be reachable from GitHub Actions runners |
| MAINTAINERS.yml schema changes | The YAML structure is stable but could add new fields. Script should be defensive (skip entries without `maintainers`/`collaborators` keys). |
| Very large MAINTAINERS.yml | Zephyr's file is large (~5000 lines). YAML parse is fast; not a concern. |
| Login case sensitivity | GitHub logins are case-insensitive. Normalize to lowercase when matching roster. |
