# feat: Host Zephyr dashboard on GitHub Pages (static snapshots)

## Overview

Host the dashboard on GitHub Pages so there is no self-hosted web service.

Because GitHub Pages is static-only, the site will be generated from **precomputed snapshot data** during a GitHub Action run (scheduled + manual), then published as static assets.

## Decisions (Confirmed)

- Data mode: **Static snapshot generated by GitHub Action** (uses PAT secret; no runtime server)
- Visibility: **Public**
- Refresh cadence: **Daily**
- Pages type: **Project site (preferred)** (`https://JimmyChen-NXP.github.io/zephyr-team-activity-dashboard/`)
  - Rationale: avoids cross-repo publishing complexity (no separate `<user>.github.io` repo required).

## Known Inputs

- GitHub username / owner: `JimmyChen-NXP`

## UX Decision (New)

- The hosted page will include an **Update data** button that opens the GitHub Actions workflow page for a manual `workflow_dispatch` run.
  - This avoids shipping any tokens to the browser and keeps GitHub Pages static-only.
  - The user will need to be logged into GitHub and have permission to run workflows.

## Open Questions

- Which repositories must be included (public only vs private org repos)?
  - This determines required PAT scopes for `DASHBOARD_GITHUB_TOKEN`.

## Constraints / Non-goals

### Constraints

- GitHub Pages cannot run Next.js server code (no Node runtime, no `app/api` routes).
- Next.js static export requires all rendered routes to be export-compatible (no request-time server rendering, no server actions, no server-only API calls).
- Anything currently depending on:
  - [src/app/api/export/route.ts](src/app/api/export/route.ts)
  - [src/app/api/github-auth/test/route.ts](src/app/api/github-auth/test/route.ts)
  will not work at runtime on Pages.

### Non-goals

- No runtime “Refresh now” / live sync on the hosted site.
- No exposing `DASHBOARD_GITHUB_TOKEN` (or any auth secrets) to the browser.

## Proposed Solution

### 1) Convert the app to a static export

Update Next.js configuration to support static export:

- Set `output: "export"` in [next.config.ts](next.config.ts)
- Ensure routes are compatible with export (no runtime server requirements)

Notes:
- For a project site (`/<repo>`), we will set `basePath` and `assetPrefix` to `/zephyr-team-activity-dashboard`.
- For a user site (fallback), base path should be root (`/`).

### 2) Replace runtime API routes with build-time generation

Replace runtime endpoints with build-time assets:

- Replace “Test connection” on hosted build:
  - Remove/disable button and show “Last generated at …” from snapshot metadata.

- Replace “Refresh now” on hosted build:
  - Replace with an **Update data** button that opens the GitHub Actions workflow page.
  - The button does not directly trigger GitHub APIs from the browser.

- Replace CSV export endpoint:
  - Option A (preferred): generate static CSV files during build and link directly.
  - Option B: generate CSV client-side from embedded snapshot JSON.

### 3) Add a snapshot generator run in GitHub Actions

Add a Node script (invoked during CI) that:

- Loads roster CSV + range presets
- Uses `DASHBOARD_GITHUB_TOKEN` secret to call GitHub API (same logic paths as live sync)
- Must run entirely in CI (token never reaches the browser)
- Produces deterministic artifacts for the static site, e.g.:
  - `public/snapshots/7d.json`
  - `public/snapshots/30d.json`
  - `public/snapshots/90d.json`
  - `public/snapshots/meta.json` (includes generatedAt, range labels)

This script should reuse existing logic where practical:

- Dashboard collection entry: [src/lib/dashboard.ts](src/lib/dashboard.ts)
- GitHub collector: [src/lib/github.ts](src/lib/github.ts)
- Range presets: [src/lib/range.ts](src/lib/range.ts)

### 4) Publish to GitHub Pages

Preferred approach: publish as a **project site** from the same repository using the built-in GitHub Pages deploy flow.

Fallback option: publish as a **user site** by pushing the static output into a separate repo named `JimmyChen-NXP.github.io`.

#### Preferred: Project site (built-in Pages deploy)

- Configure the repository’s GitHub Pages settings to deploy from **GitHub Actions**.
- Add an Action that:
  - runs snapshot generation
  - runs static build/export
  - uploads the `out/` folder using `actions/upload-pages-artifact`
  - deploys using `actions/deploy-pages`

Authentication:

- Data collection uses `DASHBOARD_GITHUB_TOKEN` (PAT) to call GitHub APIs.
- Publishing to Pages uses the repository’s built-in `GITHUB_TOKEN` (no cross-repo write needed).

URL format:

- `https://JimmyChen-NXP.github.io/zephyr-team-activity-dashboard/`

#### Fallback: User site repo (`JimmyChen-NXP.github.io`)

Because user sites require a repo named exactly `JimmyChen-NXP.github.io`, we can publish the build output from this dashboard repo into that user-site repository.

Implementation approach:

- Create (or reuse) the `<user>.github.io` repo.
- Add a GitHub Action in this dashboard repo that:
  - builds the static export (`out/`)
  - pushes that output into the `<user>.github.io` repo’s default branch

Authentication:

- Use a dedicated PAT secret (scoped to push to the user-site repo), stored as `PAGES_PUBLISH_TOKEN` in this dashboard repo.
- The GitHub API token used to collect dashboard data stays separate (`DASHBOARD_GITHUB_TOKEN`).

Notes:

- Cross-repo publishing usually requires a PAT (or deploy key) with write access to `JimmyChen-NXP.github.io`.

## Trigger Button Details

Because GitHub Pages cannot keep secrets, the in-page button should be implemented as a link/button that opens:

- `https://github.com/<owner>/<dashboard-repo>/actions/workflows/<workflow-file>.yml`

Behavior:

- Opens in a new tab.
- Copy under the button explains: “Runs the snapshot workflow (requires GitHub login). After it finishes, refresh this page.”

Notes:

- GitHub does not support a simple unauthenticated URL that *runs* a workflow; the user must click **Run workflow** in GitHub UI.

## System-Wide Impact

- **Interaction graph**: scheduled Action → snapshot generator → static export → publish → browser reads static JSON
- **Error propagation**: GitHub API failures should fail the Action run and keep last good site deployed
- **State lifecycle risks**: none at runtime; snapshots are artifacts only
- **Security**: PAT secrets remain in Actions; no tokens in client bundle

## Risks & Mitigations

- **Rate limiting**: run daily on schedule; keep manual dispatch for on-demand updates.
- **Auth confusion** (multiple tokens): document which secret is used for GitHub API reads vs Pages publishing; keep them separate.
- **Static export breakage**: verify build with `next export` mode in CI; ensure no server-only code is required for page render.
